OOP terms with definitions

Object

    The objects are the instances of the categories, that is, with the blueprint of categories we can generate many data of car categories in the program, and these data do not affect each other, each of them is independent.

Class

    In programming language, a category defines the abstract characteristics of a thing. The definition of a category contains the form of the data (attribute, Field) and the manipulation of the data (method, Method). We can also think of a class as a blueprint for a car, where we can define abstract content (i.e., attributes, methods), such as the car's license plate, the car's name, and horsepower and access to car information.

Inheritance

   Inheritance is the capability of one class to derive or inherit the properties from another class. The class that derives properties is called the derived class or child class, and the class from which the properties are being derived is called the base class or parent class. The benefits of inheritance are that it helps developers no need to create the same code again and again and add more functions that no need to revise previous codes.

Polymorphism

   Polymorphism simply means having many forms. For example, we need to determine if the given species of a supercar or not, using polymorphism we can do this using a single function.

Encapsulation

   It describes the idea of wrapping data and the methods that work on data within one unit. This puts restrictions on accessing variables and methods directly and can prevent the accidental modification of data. To prevent accidental change, an object’s variable can only be changed by an object’s method. Those types of variables are known as private variables.
   A class is an example of encapsulation as it encapsulates all the data that is member functions, variables, etc.

Type hints

   The biggest benefit of Type hints is easy code maintenance. When new members join and want to contribute code, it can save a lot of time. It is also convenient for us to provide the detection of runtime errors caused by wrong type transfer when calling Hanshu.

Static & Instance method

    For external calls, static methods can be called directly using the class, and instance methods can only be called after an instance has been created.
   For internal calls, static methods can be called within instance methods, but static methods can only call static methods, not instance methods.

SOLID


Single-responsiblity principle

   There should be only one reason for change in a class, which means that a class should only need to focus on doing one thing. This reason is that when more than two tasks need to be done in the class, once the program needs to be changed, because its coupling degree is better than one. The workload is even higher, causing difficulty in changing, so try to avoid it.

Open-closed principle

   When you add new functions, you need to be able to expand without changing the original code. In this way, you can increase the maintainability, reduce the probability of errors in the original functions, and the probability of reusability. improve.

Liskov substitution principle

   If inheritance is used, the behavior implemented by the subclass must be consistent with the behavior defined by the parent class or interface, and the subclass must be able to completely replace the parent class. If the subclass does not comply with the LSP, the user calls a function. There may be problems that do not match the expected function. The main purpose of this principle is to reduce coupling and reduce improper design.

Interface segregation principle

   When designing the interface, you should not put irrelevant methods.

Dependency Inversion Principle

   High-level modules do not depend on low-level modules, both must depend on abstract modules; abstraction cannot depend on details, details must depend on abstraction, in short, suppose I have multiple brands of light bulbs with different brightness, but as long as they meet the The specification of the lamp holder can be installed on it. In this example, the specification of the lamp holder is the abstract module.

Easier to maintain and extend the system

  The principles contained in SOLID are to clean up the code bug of the software by causing the programmer to refactor the software source code, so as to make the software clear and readable and can be applied when it is extensible. SOLID is typically used in test-driven development and is an important part of the basic principles of agile development and adaptive software development.

Design Patterns


    There are two main purpose for design patterns

    Common platform for developers

    Design patterns provide a standard terminology system specific to a specific situation. For example, the singleton design pattern means using a single object, so that all developers familiar with the singleton design pattern can use the single object, and in this way can tell each other that the program is using the singleton pattern.
   Best practice

   Design patterns have evolved over a long period of time and they provide the best solutions to common problems faced during software development. Learning these patterns helps inexperienced developers learn software design in an easy and fast way.
Six principles of design patterns

    1. Open Close Principle

    The open-closed principle means: open for extension, closed for modification. When the program needs to be expanded, the original code cannot be modified to achieve a hot-plug effect. In short, it is to make the program expandable, easy to maintain and upgrade. To achieve this effect, we need to use interfaces and abstract classes, which we will mention in the specific design later.
    2. Liskov Substitution Principle

    The Liskov Substitution Principle is one of the basic principles of object-oriented design. The Liskov Substitution Principle states that wherever a base class can appear, a subclass must appear. LSP is the cornerstone of inheritance and reuse. Only when the derived class can replace the base class and the function of the software unit is not affected, the base class can be truly reused, and the derived class can also add new ones on the basis of the base class. behavior. The Liskov Substitution Principle is a complement to the Open-Closed Principle. The key step to realize the open-closed principle is abstraction, and the inheritance relationship between the base class and the subclass is the concrete realization of abstraction, so the Liskov substitution principle is the specification of the concrete steps to realize abstraction.
    3. Dependence Inversion Principle

    This principle is the basis of the open-closed principle, concrete content: programming against interfaces, relying on abstraction rather than concreteness.
    4. Interface Segregation Principle

    What this principle means is that it is better to use multiple isolated interfaces than to use a single interface. It has another meaning: reduce the coupling between classes. It can be seen that, in fact, the design pattern is a software design idea that starts from a large-scale software architecture and is easy to upgrade and maintain. It emphasizes reducing dependencies and coupling.
    5. The Law of Demeter, also known as the Principle of Least Knowing (Demeter Principle)

    The principle of least-knowing means that an entity should interact with other entities as little as possible, so that the functional modules of the system are relatively independent.
    6. Composite Reuse Principle

    The principle of composition reuse means: try to use composition/aggregation instead of inheritance.
